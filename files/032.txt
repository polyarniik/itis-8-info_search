        </span></div> </div></div> <!-- --> <!-- --></div></div> <!-- --> <div class="tm-article-body" data-gallery-root="" lang="ru"><div></div> <div id="post-content-body"><div><div class="article-formatted-body article-formatted-body article-formatted-body_version-2"><div xmlns="http://www.w3.org/1999/xhtml"><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/d15/b28/1fa/d15b281faf477d64c415afb72a76aa3b.png" height="439" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/d15/b28/1fa/d15b281faf477d64c415afb72a76aa3b.png" width="780"><figcaption></figcaption></img></figure><div class="persona" persona="true"><img class="image persona__image" data-blurred="true" data-src="https://habrastorage.org/getpro/habr/upload_files/97a/792/4b6/97a7924b62b9be7f129abd1951ae7b53.jpg" persona="true" src="https://habrastorage.org/r/w780q1/getpro/habr/upload_files/97a/792/4b6/97a7924b62b9be7f129abd1951ae7b53.jpg"/><h5 class="persona__heading" persona="true">Автор статьи: Андрей Поляков</h5><p>Старший разработчик в Unlimint</p></div><p>Хабр, добрый день! Что первое приходит в голову, когда вы слышите “ETL”? Скорее всего airflow?</p><p>По сути airflow - это стандарт де-факто в мире обработки и трансформации данных. Но в случае если в разработка ведется на java, то тогда придется либо переучивать разработчиков на python, либо нанимать еще python разработчиков. В большинстве случаев хочется вносить минимум изменений в процесс разработки и тех. стек. Решение есть! Фреймворк Spring Cloud предоставляет DataFlow - фреймворк для организации ETL в spring среде.</p><h3>Архитектура</h3><p>Фреймворк DataFlow состоит из двух основных компонентов:</p><ul><li><p>Data Flow Server</p></li><li><p>Skipper Server  (шкипер? 🙂)</p></li></ul><p>Высокоуровневая архитектура показана на схеме ниже:</p><p> </p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/e0a/348/105/e0a348105e51904735de4a16836f578d.png" height="310" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/e0a/348/105/e0a348105e51904735de4a16836f578d.png" width="582"/><figcaption></figcaption></figure><p>Основная точка входа для взаимодействия с DataFlow - это REST API сервиса Data Flow Server. Web Dashboard и Data Flow Shell также взаимодействуют с Data Flow Server. Система может работать на различных платформах: Cloud Foundry, Kubernetes или на вашем локальном компьютере.</p><p>Обратите внимание, что и Data Flow Server, и Skipper Server хранят данные (и свое внутреннее состояние) в своей собственной РСУБД. Можно подключить одну физическую СУБД к обоим сервисам, так как каждый из них создает свой собственный namespace/schema внутри СУБД. По умолчанию каждый из сервисов использует встроенную базу данных H2.</p><p>Давайте познакомимся поближе с Data Flow Server. Он отвечает практически за все 🙂</p><ul><li><p>парсинг задач (job-ов) для стриминга и батчинга на основе своего собственного DSL (о нем чуть позже);</p></li><li><p>валидация, сохранение описаний задач стриминга и батчинга;</p></li><li><p>регистрация артефактов (таких как jar файлы и докер образы), которые испльзуются в шагах пайплайнов;</p></li><li><p>деплой задач;</p></li><li><p>планирование работы задач;</p></li><li><p>получение истории выполнения стриминговых и батч-задач;</p></li><li><p>конфигурирование свойств задач (inputs/outputs, начальное количество инстансов, разделение данных и т.д.);</p></li><li><p>делегирование деплоя стриминговых задач сервису Skipper;</p></li><li><p>аудит (таких событий как создание стрима, деплой и уничтожение батч-задач);</p></li><li><p>и многое друге.</p></li></ul><p>Впечатляющий список, не правда ли?</p><p>Другой компонент системы - Skipper Server отвечает в основном за выполнение стриминговых задач.</p><ul><li><p>Развертывание потоков на одной или нескольких платформах.</p></li><li><p>Обновление и откат потоков на одной или нескольких платформах с использованием стратегии синего/зеленого деплоя.</p></li><li><p>Хранение истории файла манифеста каждого потока (который представляет окончательное описание того, какие приложения были развернуты).</p></li></ul><h3>Строим ETL</h3><p>Обычно ETL пайплайны в SCDF (Spring Cloud DataFlow) состоят из 3-х компонентов:</p><ul><li><p>Source - извлечение данных;</p></li><li><p>Processor - преобразование данных;</p></li><li><p>Sink - загрузка.</p></li></ul><p>Давайте рассмотрим пример ETL-пайплайна:</p><ol><li><p>Данные получаем по HTTP (source).</p></li><li><p>Преобразуем и насыщаем данные (processor).</p></li><li><p></p><ol><li><p>Сохраняем данные в лог (sink).</p></li><li><p>Проведение анализа данных (processor).</p></li></ol></li><li><p>Сохраняем данные в файл  (sink).</p></li></ol><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/e17/5a4/bbc/e175a4bbc16cfe01fb537beeefdafc11.png" height="291" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/e17/5a4/bbc/e175a4bbc16cfe01fb537beeefdafc11.png" width="1146"/><figcaption></figcaption></figure><p>Для передачи данных между блоками ETL‑пайплайна используется так называемый <strong>messaging middleware </strong>— промежуточный слой брокера сообщений. Spring Cloud Data Flow поддерживает два брокера сообщений в качестве messaging middleware — Apache Kafka and RabbitMQ.</p><p>Для запуска Spring Cloud Data Flow удобнее всего использовать docker‑compose (хотя это далеко не единственный способ!). Обратимся к официальной документации проекта SCDF и получим ссылки для скачивания файлов <code>docker-compose</code>:</p><pre><code>curl https://raw.githubusercontent.com/spring-cloud/spring-cloud-dataflow/main/src/docker-compose/docker-compose.yml -o docker-compose.yml
curl https://raw.githubusercontent.com/spring-cloud/spring-cloud-dataflow/main/src/docker-compose/docker-compose-&lt;broker&gt;.yml -o docker-compose-&lt;broker&gt;.yml
curl https://raw.githubusercontent.com/spring-cloud/spring-cloud-dataflow/main/src/docker-compose/docker-compose-&lt;database&gt;.yml -o docker-compose-&lt;database&gt;.yml</code></pre><p>Здесь:</p><p><code>&lt;broker&gt;</code> — это брокер сообщений (либо kafka, либо rabbitmq).</p><p><code>&lt;database&gt;</code><strong> </strong>— база данных, которая будет использоваться Data Flow Server и Skipper Server (может быть postgres, mariadb или mysql).</p><p>Запускаем скачанные <code>docker-compose</code>, перед запуском необходимо установить через переменные окружения версии сервисов SCDF, которые будут использоваться. Например:</p><pre><code>set DATAFLOW_VERSION=2.10.1&amp;
set SKIPPER_VERSION=2.9.1</code></pre><p>После того как сервисы в docker‑compose запустились, они будут доступны по следующим портам:</p><div><div class="table"><table><tbody><tr><td><p align="left"><strong>Host ports</strong></p></td><td><p align="left"><strong>Container ports</strong></p></td><td><p align="left"><strong>Description</strong></p></td></tr><tr><td><p align="left">9393</p></td><td><p align="left">9393</p></td><td><p align="left">На этом порту работает Data Flow server. Можно открыть дашборд по адресу <a href="http://localhost:9393/dashboard">http://localhost:9393/dashboard</a>  или взаимодействовать с  REST API по адрес <a href="http://localhost:9393/">http://localhost:9393</a></p></td></tr><tr><td><p align="left">7577</p></td><td><p align="left">7577</p></td><td><p align="left">На этом порту работает Skipper server. Можно повзаимодействовать с Skipper REST API по адресу <a href="http://localhost:7577/api">http://localhost:7577/api</a></p></td></tr></tbody></table></div></div><p>Важное примечание:</p><p>Сейчас Spring Cloud Data Flow по умолчанию использует Java 8 для запуска приложений. Если вы хотите запускать приложения, собранные с Java 17, нужно установить значение переменной окружения <code>BP_JVM_VERSION</code>:</p><pre><code>export BP_JVM_VERSION=-jdk17</code></pre><p>Также вам скорее всего понадобится возможность отладки приложений и пайплайнов. К сожалению, это не такая простая задача, но в целом очень подробно описана в документации: <a href="https://dataflow.spring.io/docs/installation/local/docker-customize/#debug-stream-applications"><u>Customizing Docker Compose</u></a>. Также там же описывается, как добавить zipkin для распределенной трассировки ваших пайплайнов. </p><p>Давайте начнем знакомство с веб‑ui системы SCDF.</p><p>Здесь мы видим несколько разделов:</p><ul><li><p>приложения</p></li><li><p>потоковые пайплайны</p></li><li><p>задачи (tasks)</p></li><li><p>управление и аудит</p></li></ul><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/1f1/1c1/030/1f11c1030d9e1c400b619e1293358b5a.png" height="423" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/1f1/1c1/030/1f11c1030d9e1c400b619e1293358b5a.png" width="1600"/><figcaption></figcaption></figure><p>Самая первая вкладка — <strong>приложения</strong>. Здесь мы можем загружать «приложения» — составные блоки нашего будущего ETL‑пайплайна.</p><p>Нажмем на кнопку «добавить приложение». Открывается окно с выбором способа добавления приложения:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/2ec/5dd/328/2ec5dd328c2fbddfed506693574d8816.png" height="562" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/2ec/5dd/328/2ec5dd328c2fbddfed506693574d8816.png" width="1600"/><figcaption></figcaption></figure><p>В самом простом случае нас интересует вкладка «зарегистрировать одно или несколько приложений». Указываем название приложения (скорее всего будет совпадать с названием шага в ETL‑пайплайне, которому оно соответствует), выбираем тип приложения и URI. Обратите внимание, что также нужно указать Metadata URI с мета‑данными вашего приложения.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/485/5d2/f01/4855d2f010975e071528e198b9227359.png" height="569" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/485/5d2/f01/4855d2f010975e071528e198b9227359.png" width="1319"/><figcaption></figcaption></figure><p>Добавили приложение? Ок, двигаемся дальше и приступаем к созданию пайплайна.</p><p>Для этого нужно перейти в раздел “Потоки” и нажимаем на кнопку “создать потоки”.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/db0/04a/02c/db004a02cac49a4801d434c4e7503a0b.png" height="785" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/db0/04a/02c/db004a02cac49a4801d434c4e7503a0b.png" width="1600"/><figcaption></figcaption></figure><p>Есть два способа описать ETL пайплайн:</p><ul><li><p>в визуальном редакторе на форме;</p></li><li><p>с помощью DSL‑языка.</p></li></ul><p>Неважно, какой из способов вы выберете, второй подход будет автоматически добавлен тоже.</p><p>Например:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/955/99d/fc6/95599dfc6e657f3657df83815be41217.png" height="760" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/955/99d/fc6/95599dfc6e657f3657df83815be41217.png" width="1600"/><figcaption></figcaption></figure><p>Поток был создан с помощью визуального редактора и при этом автоматически продублирован в виде DSL:</p><pre><code class="java">STREAM-1=http | header-enricher | log
:STREAM-1.header-enricher &gt; transform | groovy | file</code></pre><p>Нажимаем “создать потоки” и поток отобразится в таблице.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/088/e93/aa2/088e93aa23853730d8d6c01305a5016c.png" height="848" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/088/e93/aa2/088e93aa23853730d8d6c01305a5016c.png" width="1600"/><figcaption></figcaption></figure><p>Обратите внимание: в таблице отобразились два отдельных потока. При этом конвейер данных у нас один (т.е. оба потока связаны друг с другом и данные из одного передаются в другой):</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/39e/dff/12e/39edff12eb2398a6812d19ed1542a0e3.png" height="268" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/39e/dff/12e/39edff12eb2398a6812d19ed1542a0e3.png" width="1124"/><figcaption></figcaption></figure><p>Дальше можно запускать сам процесс!</p><p>Предположим, что поток данных пошел и наш поток может начать их принимать и обрабатывать.</p><p>Переходим к нашему потоку и видим, что он еще не задеплоен.</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/5bf/fe0/9c2/5bffe09c2779588e68402a44da61e791.png" height="542" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/5bf/fe0/9c2/5bffe09c2779588e68402a44da61e791.png" width="1600"/><figcaption></figcaption></figure><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/73c/969/67d/73c96967da6ff89666906cbfd4baf83a.png" height="542" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/73c/969/67d/73c96967da6ff89666906cbfd4baf83a.png" width="1600"/><figcaption></figcaption></figure><p>Первое, что нужно сделать — это задеплоить поток. Нажимаем кнопку «задеплоить» и переходим на форму, на которой необходимо указать параметры развертывания для потока целиком и для каждого из его компонентов:</p><figure class="full-width"><img data-src="https://habrastorage.org/getpro/habr/upload_files/216/ec5/ac2/216ec5ac2be84741ec984b1cfcd81613.png" height="867" src="https://habrastorage.org/r/w1560/getpro/habr/upload_files/216/ec5/ac2/216ec5ac2be84741ec984b1cfcd81613.png" width="1274"/><figcaption></figcaption></figure><p>Отлично! Ваш поток развернут и готов к работе.</p><p><strong>Как проверить работу пайплайна?</strong></p><p>Для созданного примера достаточно, чтобы по указанному http адресу появились данные, которые наш ETL пайплайн сможет получить и обработать. В конце можем проверить успешность пайплайна, если создался файл с преобразованными данными и создалась запись в логе.</p><h3>Пару слов про свои собственные приложения</h3><p>В рассмотренном выше примере мы использовали стандартные приложения, которые уже сделаны за нас и их можно скачать. А что делать, если нужно выполнить логику, для которой не подходят уже существующие приложения? Написать свою.</p><p>Давайте посмотрим на примере приложения‑processor (у которого есть ендпойнт для получения входных данных и для проброса данных дальше).</p><p>Такое приложение можно уместить в одном файле. Например:</p><pre><code class="java">@EnableBinding(Processor.class)
@SpringBootApplication
public class MyProcessorApplication {

 @Autowired
SomeComponent someComponent;

 public static void main(String[] args) {
  SpringApplication.run(MyProcessorApplication.class, args);
 }

 @StreamListener(Processor.INPUT)
 @SendTo(Processor.OUTPUT)
 public Object processMessage(Message&lt;?&gt; message) {
  System.out.println(message);
  MessageHeaders header = message.getHeaders();
   byte[] body = (byte[]) message.getPayload();
   // do some logic and transformations
   someComponent.doSomething(header, body);
  return message;
 }
}
</code></pre><p>После того, как код приложения написан, можно зарегистрировать его в Data Flow Server через веб‑ui или через shell. Так как через ui мы уже видели интерфейс, приведу пример, как зарегистрировать через shell:</p><pre><code>app register --name myProcessor --type processor --uri maven://com.example:my-processor:jar:0.0.1-SNAPSHOT</code></pre><p>После этого при создании потоков можно использовать наше только что зарегистрированное приложение наравне с другими.</p><p><strong>Подведем итоги:</strong> для java‑мира есть довольно удобная, мощная и гибкая альтернатива для airflow и можно ее использовать для построения ETL‑пайплайнов, конвейеров потоковой обработки данных и процессов обработки тяжелых единоразовых задач.</p><p>Ну и напоследок приглашаю всех на <a href="https://otus.pw/xtiT/">бесплатный урок</a>, где мы рассмотрим такой принцип проектирования программного обеспечения, как «Соглашения по конфигурации», Поговорим про «Скриптовый Spring» и затронем не менее важную тему «Безопасного программирования».</p><ul><li><p><a
